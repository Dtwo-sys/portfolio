WSPR (Weak Signal Propagation Reporter) is an amateur radio mode designed for one narrow purpose: finding out how well signals are travelling. Transmissions are very low power — typically a watt or less — and very short. Each transmission encodes a callsign, grid locator, and power level, then gets decoded and logged by receivers worldwide. The results land at [wspr.live](https://wspr.live), a public database of reception reports updated continuously. It is an efficient way to read the ionosphere.

I run WSPR from my station (G5OHM), using WSJT-X software alongside an HF transceiver. The data is there on WSPRnet and PSKReporter, available to look at any time. What I wanted was to see it inside Home Assistant alongside everything else on my dashboard, without switching between applications.

There was no existing integration. I built one.

## What the integration does

Two sensors are created:

- `sensor.wspr_live_rx_spots` — stations I have received
- `sensor.wspr_live_tx_spots` — stations that have received me

Every 30 to 60 minutes the integration queries the WSPR.live API, processes the returned JSON, and updates both sensors. Each spot entry includes the callsign, signal-to-noise ratio, frequency band, distance in kilometres, and country. The country is resolved from the callsign prefix using a lookup table; the band (20m, 40m, and so on) is mapped from the raw frequency value using a static dictionary.

Home Assistant input helpers track the furthest received and transmitted distances, updating automatically whenever a new record is set. The dashboard I put together shows a live table of spots, an SNR trend over time, and the running distance records for each direction.

Everything runs locally. The only external call is to the WSPR.live API.

## How it was built

The data flow is straightforward: a polling coordinator fetches fresh data on the configured interval, processes it, and makes the result available through the standard Home Assistant entity model. The data structures involved are not complicated. Getting the Home Assistant integration architecture right — config entries, coordinators, entity registration — takes longer to understand than the core logic.

I am not a software developer. My background is in electronics and instrumentation. Building something like this would previously have meant months of learning Python seriously or finding someone else to write it. What changed is that AI tools — I used Claude and ChatGPT across different stages — can generate structural scaffolding, explain what error messages mean, and suggest how to fix them. The decisions about what to display, how to structure the data, and when something was working correctly were mine. The AI kept me moving through the parts I would otherwise have got stuck on.

The code is on GitHub: [Dtwo-sys/wspr\_live](https://github.com/Dtwo-sys/wspr_live).

If you run WSPR and have Home Assistant, it should work with configuration similar to any other custom component. If you find problems or have suggestions, raise an issue on the repository.
